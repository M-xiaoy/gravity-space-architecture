# 重力空间架构 · 研究方案

> 核心问题：一个 token 在概率空间中的位置（logit 值）包含的信息量，
> 远少于它在共振空间中的位置（多源证据强度的分布）所包含的信息量。
>
> **多样性维度是 softmax 完全丢弃的信息。**
> ——这是整个架构唯一需要实验验证的核心命题。

---

## 一、成熟度评估

**直接回答：架构还未达到可落地的程度。但这不是因为设计得不够细——是因为缺少一次验证闭环。**

Phase 1 → 4 完成了：
- ✅ 理论自洽（逻辑链条闭合）
- ✅ 组件设计（接口、存储、预测、归因）
- ✅ 数学映射（双曲空间、Neural ODE、Modern Hopfield）
- ✅ 工程估算（3MB–53MB 存储，400 万参数预测头）

缺少的：
- ❌ 核心命题的实验验证（多样性信息是否真的被浪费了？）
- ❌ 引力扰动 vs 标准 attention 的 A/B 测试
- ❌ 预测编码的收敛曲线
- ❌ 可运行的代码

**研究策略：先把核心命题从假说变成可观测事实，再设计完整工程。** 否则你花两周搭了一个完整的架构，然后发现「多样性维度对输出质量没有显著影响」——那就白做了。

## 二、原型验证设计：Gravity Reranker

### 核心理念

不做两层分离、不修改 Transformer 内部、不搭双曲空间。

只做一件事：**证明多样性信息是存在的、可提取的、对决策有价值的。**

### 架构

```
输入问题
  ↓
LLM 多次采样（T 条不同的推理路径）
  ↓
对每条路径的每一步，记录：
  ├── 该步生成的 token
  ├── 该步的概率分布 (logits)
  └── 各注意力头对输入 token 的权重分布
  ↓
路径合并 → 每个候选答案的统计量：
  ├── 平均 log-prob（标准做法）
  └── 跨路径支持多样性（新做法）
  ↓
Gravity Score = avg_log_prob × (1 + α × diversity)
  ↓
按 Gravity Score 重排候选答案 → 输出
```

### 这是你在做什么

不是真的在改模型——是在证明：

**「同样的概率值背后，可能有完全不同的证据多样性结构。softmax 看不到这个结构，但我们可以绕过 softmax，在外部把多样性信息加回来。」**

如果这个 reranker 在任何任务上表现出优势——哪怕只有 3-5% 的提升——那就说明核心命题成立。然后你才有底气去碰真正的两层架构。

## 三、实验任务设计

### 任务选型原则

避免选 LLM 已经做得特别好的任务（常识问答、摘要）——天花板太低。
选需要 **多源证据交叉验证** 的任务——这才是重力空间理论上的优势区。

### 三个候选任务

**任务 A：矛盾文本推理（推荐首选）**

给模型两段互相矛盾的文本，问一个需要交叉验证才能回答的问题。

示例：
```
文本1：小明养了一只猫和一只狗。猫叫咪咪，狗叫旺财。
        每天早上咪咪会在窗台上等日出。

文本2：小明的宠物中只有狗会在早上叫醒他。
        猫通常要睡到中午才会活动。

问题：咪咪早上会在哪里？
  A. 窗台上
  B. 窝里睡到中午
  C. 不确定
```

正确答案是 C（不确定）——因为两条证据路径对「咪咪早上在哪」给出了矛盾的结论。标准 LLM 会倾向于选概率最高的 A（因为"窗台等日出"是更具体的、看起来更合理的信息），但多样性分析能看到：支持 A 的路径只有一条，支持 B 的路径也有一条，互不相容 → 不确定性应该高。

**任务 B：多源事实核查**

给模型一篇文章，其中包含来自多个来源的事实声明。部分来源可靠、部分不可靠。模型需要判断哪些声明是可信的。

示例：
```
来源A（权威期刊）："全球平均气温在过去50年上升了1.2°C"
来源B（社交媒体）："全球平均气温上升了5°C，科学家都在隐瞒真相"
来源C（政府报告）："过去50年气温呈上升趋势，但具体幅度有待确认"

问题：关于全球变暖，以下哪个说法最可信？
```

多样性在这里表现为：支持「上升约1°C」的证据路径（A的权威 + C的保守确认），比支持「上升5°C」的单一路径（B的夸张）更丰富。

**任务 C：创造性写作的选择多样性**

给模型一个创作任务（续写故事/写对联/起名字），要求从多个候选中选择最佳方案。

这个任务上标准 LLM 倾向于选最常见的、概率最高的方案（无趣但安全）。多样性维度可以帮它识别「哪个方案被多个不同的推理路径同时支持」——即哪个方案在多个评估标准（语法、风格、主题契合）上都通过了。

### 任务选择建议

**先做任务 A**——矛盾文本推理。因为：
1. 数据可以合成，不需要人工标注
2. 正确答案是确定的（A/B/C）
3. 预期效果好——矛盾场景下标准 LLM 表现最差
4. 单次实验就能得到明确的信号

## 四、实验流程

### Step 1：数据合成

用 LLM 生成 200 条矛盾文本对，每条带 3 个选项（1个正确，2个干扰）。
每条约 200-300 字。

### Step 2：基线

对每条测试样例：
1. 把完整文本输入 LLM
2. 直接采样答案
3. 记录选中的答案和概率
4. 计算基线准确率

### Step 3：Gravity Reranker

对每条测试样例：
1. 输入文本 → 让 LLM 生成 T 条不同的推理路径（temperature 采样，T=5）
2. 对每条路径，提取每一步的注意力分布
3. 对每个候选答案，计算：
   - avg_log_prob：所有指向该答案的路径步的 log-prob 均值
   - diversity：跨路径的注意力分布差异度（余弦距离的均值）
   - gravity_score = avg_log_prob × (1 + α × diversity)
4. 按 gravity_score 重排候选答案
5. 计算重排后的准确率

### Step 4：对比

| 指标 | 基线 | Gravity Reranker |
|------|------|-----------------|
| 准确率 | xx% | xx% |
| 不确定性拒绝率 | xx% | xx% |
| 错误自信率 | xx% | xx% |

### Step 5：超参扫描

在验证集上扫描 α（多样性权重），范围 [0, 2]，步长 0.1。
找到最优 α 后在测试集上验证。

## 五、技术可行性

### 你需要的东西

| 组件 | 已有 | 需要写 |
|------|-----|-------|
| LLM 推理 | ✅ Ollama（qwen/llama） | — |
| 多路径采样 | — | 循环调用 + 温度控制 |
| 注意力提取 | — | 需要模型支持输出 attention weights |
| 多样性计算 | — | 几十行 numpy |
| 数据生成 | — | LLM 造数据 + 人工审 |
| 评估流水线 | — | 自动计算 + 对比 |

### 注意力提取的可行性

Ollama 加载的模型不一定暴露 attention weights。需要确认：
- llama.cpp 支持 `--log-format json` 输出部分 log
- 或者直接通过 transformers 库加载 HuggingFace 模型
- 或者退而求其次：用 logits 在不同 temperature 下的多轮采样近似「路径多样性」

**备选方案**：如果 attention weights 无法直接获取，用 logits 分布的熵作为多样性的代理度量。虽然精度不如注意力分布，但足以验证核心命题。

### 工程复杂度估算

| 阶段 | 时间估算 | 产物 |
|------|---------|------|
| 数据生成脚本 | 2 小时 | 200 条矛盾文本 |
| 基线评估 | 1 小时 | 基线准确率 |
| Reranker 实现 | 3 小时 | gravity_score 计算 |
| 完整实验运行 | 2 小时 | 对比结果 |

总计：约 9 小时。一个周末能跑完。

## 六、验证标准

### 通过标准

Gravity Reranker 在矛盾推理任务上，满足以下任一条件即视为 **核心命题得到实验支持**：

1. **准确率提升 ≥ 3%**（绝对百分点）
2. **不确定性拒绝率提升**——模型在矛盾场景下更频繁地输出「不确定」，而不是自信地选一个错的
3. **错误自信率降低**——选错时的平均概率降低（错了但没那么自信）

### 优秀标准

1. 所有三项指标同时改善
2. 优势在跨 temperature 设置下保持稳定（T=3 / T=5 / T=10 都有效）
3. 多样性权重的存在显著提升 → 证明多样性信息确实被基线浪费了

### 失败标准

1. Gravity Reranker 不比基线好
2. 或者好但效果不稳定（T 变了效果就没了）
3. 或者多样性权重最优值为 0（即多样性没有信息价值）

**如果失败：**
- 核心命题不成立，整个架构需要重新审视
- 但可能性较低——注意力分布跨头的方差在多个领域都被证明携带有用信息（BERTology 文献充分支持）
- 更可能的结果是：有效，但幅度不大（2-5%）
