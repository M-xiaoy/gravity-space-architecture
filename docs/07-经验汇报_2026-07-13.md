## 二、遇到的错误及解决

### 1. 健康检查假阴性

**现象：** `kb_ctl.py status` 显示"端口占用但无响应"，实际服务运行正常。

**根因：** `is_server_healthy()` 用 httpx 检查 `127.0.0.1:8765/status`。httpx 默认 `trust_env=True`，在 Windows 上会读取系统代理设置，本机 Clash/V2Ray 代理设为 `127.0.0.1:80`。请求被发给了代理而不是知识库服务。

**解决：** 将 httpx 替换为 `urllib.request`（不继承系统代理环境变量）。

**教训：** 本地系统工具的健康检查不要走 HTTP 客户端库的默认代理设置。`urllib.request` 在 Windows 上不走系统代理，更适合本地检查。

### 2. uvicorn 孤儿进程

**现象：** 反复 `kb_ctl.py restart` 后端口被残留进程占用，新进程绑不上。

**根因：** `kb_ctl.py start` 通过 `subprocess.Popen` + `CREATE_NEW_PROCESS_GROUP` 启动 uvicorn。exec 工具超时（SIGKILL）杀了 ctl.py 进程，但 uvicorn 作为子进程独立存活，变成孤儿继续占端口。

Windows 上 uvicorn 端口冲突时不报 `Address already in use`，而是静默打印 `startup complete` 但不实际绑定——Uvicorn GitHub Issue #435 确认的 bug。

**解决：** `kb_ctl.py stop` 增加两步清理：先读 PID 文件杀，再 `find_uvicorn_pids()` 扫描残留。

**教训：** Windows 上子进程管理不可靠。父进程死了子进程不一定死。加上 PID 文件 + 进程扫描双保险才能保证端口清理干净。

### 3. Session 管理 TypeError

**现象：** `session.is_expired()` 报 `TypeError: 'bool' object is not callable`。

**根因：** `is_expired` 同时加了 `@property` 装饰器但方法签名接受 `timeout_seconds` 参数。`@property` 不能带参数，Python 把方法调用结果当成属性返回，再调用时报错。

**解决：** 去掉 `@property`，调用时显式传参。

### 4. 首次聚类未持久化 cluster_id

**现象：** 重启后发现所有球体的 `cluster_id` 为 -1，场域列表回退到旧标签名。

**根因：** 启动时首次聚类后，更新了内存中 Sphere 对象的 `cluster_id`，但未调用 `sphere_store.save()` 持久化。下次重启时 `cluster_engine.load()` 成功（聚类中心已存盘），但球体的 `cluster_id` 丢了。而后续的"标签均值回退"代码又把 field_detector 覆盖回了旧标签质心。

**解决：** 启动加载聚类状态后，用 `cluster_engine.predict()` 重新分配所有活跃球体的 `cluster_id`。同时给聚类加载分支加上条件判断——有聚类状态时跳过标签均值回退。

**教训：** 内存状态变更后如果不持久化，重启即丢失。加载聚类状态时不仅要恢复 centroids，还要恢复每个球体的归属。

### 5. 场域聚焦过滤结果不匹配

**现象：** 传入 `field_focus: "技术笔记"` 后，返回的依然是所有场域混合的结果。

**根因：** 场域聚焦逻辑在检索层（retriever.py）用 `source_type` 做过滤，而聚类后场域名变成了 `簇0/簇1/簇2`。过滤条件不匹配。

**解决：** 将聚焦过滤从 `source_type` 匹配改为 `cluster_id` 匹配。提取 `field_focus` 中的簇 ID 数字，与球体的 `cluster_id` 字段对比。

### 6. MiniBatchKMeans 聚类不稳定

**现象：** 1459 条数据跑 MiniBatchKMeans，三个聚类中心余弦距离 0.87-0.90（几乎重合），所有球体分配到同一个簇。

**根因：** MiniBatchKMeans 专为大数据集设计（十万级以上），对小数据集（~1500 条）的批次采样不稳定。三个簇都收敛到了数据的稠密区域中心。

**解决：** 更换为 `sklearn.cluster.KMeans`（全量 Lloyd 算法）。全量 KMeans 每次迭代都用全部数据更新质心，对 1500 条数据仅需 ~20ms。

**教训：** 不要因为"名字叫 MiniBatch"就觉得它更好。全量 KMeans 在数据量小的时候更快更稳定。

## 三、错误类型分布

| 类型 | 数量 | 原因 |
|------|------|------|
| 环境/工具链 | 2 | httpx 代理劫持、uvicorn 孤儿进程 |
| 类型/逻辑 | 3 | is_expired @property 误用、未持久化、过滤条件不匹配 |
| 算法选择 | 1 | MiniBatchKMeans 不适用当前数据量 |

6 个错误中，4 个是**写代码时就能避免的**（代理、property、持久化、过滤条件），2 个是**系统性知识缺陷**（uvicorn Windows bug、算法选型）。
