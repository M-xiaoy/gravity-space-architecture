# 1 · 核心命题与架构哲学

> 2026-07-11 定稿
> 这个文件定义了重力空间架构的「为什么」——所有组件设计都从这里导出。

---

## 0. 一句话定位

**不是让模型更准确，而是让它更多样、有潜在思维、更像人。**

重力空间架构的目标不是最小化 loss 或最大化准确率——而是在保持推理质量的前提下，让系统内部能够维持多条互不相容的推理路径同时存在，允许张力和不确定性作为认知的正常组成部分。

---

## 1. 核心命题

**多样性维度是 softmax 完全丢弃的信息。**

标准 Transformer 的 softmax 层将 logits 归一化为概率分布——它强迫所有可能性之和为 1。但「一个答案有多大概率被选中」和「支撑这个答案的证据有多多样」，是两个正交的维度。

重力空间架构的核心假设是：**同样的概率值背后，可能有完全不同的证据多样性结构。** 多样性信息可以被提取、度量、并纳入决策过程——而这正是「更像人」的关键：人在做决策时不只是看哪一个选项概率最高，还会考虑多个独立证据来源的相互支撑程度。

**验证方法：** Gravity Reranker（见 [6-实验记录.md](6-实验记录.md)）

## 2. 与标准 ML 范式的关键分歧

| 维度 | 标准 ML | 重力空间 |
|------|---------|---------|
| 优化目标 | loss → 0，准确率 → 100% | 保持多样性，维持张力 |
| 对误差的态度 | 缺陷，需消除 | 新思想的触发信号 |
| 收敛 | 终点，是好事 | 需避免——收敛 = 多样性丧失 |
| softmax | 核心操作 | 需要「绕过」的对象 |
| 推理路径 | 单条最佳路径 | 多条并行，相互竞争但共存 |
| 空闲时间 | 无意义 | 核心维护时段（复盘/归档/清理） |
| 噪声 | 数据质量问题 | 重复旧信息抑制新内容的外在表现 |

## 3. 设计原则

### 3.1 清醒默认
系统有「状态」。需要维护，不是无代价的。认知状态管理器决定什么时候该干什么——不是在所有时刻都全力响应。

### 3.2 张力优先于一致
两条矛盾的推理路径同时存在是架构的正确工作方式，不是 bug。系统不强迫自己「选一条」。

### 3.3 外部反馈是地形修正信号
用户不同意某条推理 → 系统进入解释模式（复现路径 + 影响因素）→ 用户坚持不同意 → 该推理标为 `pending_revision` → 下次 idle 时纳入地形更新。

### 3.4 变更要有自我觉察
系统需要能够指向自己过去的状态并描述差值。当它的输出因内部更新而改变时，应该能回答：「我之前是这么想的，但现在因为 X 原因调整了。」

### 3.5 Idle 不是关机，是呼吸
系统在空闲时自动执行三件事：
- **噪音清理** — 删除冗余的弱连接
- **内容复盘** — 重放近期轨迹，巩固重要模式  
- **记忆归档** — 去冗余、建索引、压缩存储

Idle 是 best-effort 维护，可以随时中断，中断内容在下一轮 idle 补偿。

## 4. 重复申明的边界

- 这个架构不是通用 Transformer 改进方案。它的目标用户是**一个人**的系统。
- 伦理上这是在探索「如何让系统更像人」的计算模型。这是认知探索，不是部署伦理问题。但需要意识到这条路的敏感性。
- 如果架构输出了一条用户无法接受的推理路径——**那是架构的正确工作结果**，不是 bug。

## 5. 外部参考

- 溪流计划 —— 本架构从「持续性存在」的设计哲学中汲取了思路。但架构本身的传感器/管道需要重新设计，不直接复用溪流代码。
- Modern Hopfield Networks —— 能量函数的吸引子盆地与重力场的「引力盆地」同构。
- Neural ODE —— 连续深度网络的动力学为思考层的持续更新提供了数学框架。
- 预测编码理论 —— Phase 4 预测头的认知科学基础。
